01Problema
Multi-repo + IA
Por que IA sozinha não resolve

Os agentes precisam ver o ecossistema inteiro.
Os repositórios não conversam entre si.

Modelos de linguagem trabalham com o contexto que recebem. Quando cada serviço vive em seu próprio repositório, isolado, o agente nunca tem o todo — apenas pedaços.

A consequência prática: as respostas da IA são tão fragmentadas quanto o ecossistema. Bom o suficiente para um arquivo, frágil cruzando fronteiras de repositório.

Nrepos
Dezenas de repositórios independentes, em linguagens distintas, com cadências próprias — e nenhuma visão unificada.
0contexto
É o que um agente vê sobre os outros repositórios quando opera apenas dentro de um deles.

02Solução
O que é um submódulo
Mecânica básica

Um submódulo é um ponteiro de commit dentro de outro repositório.

# .gitmodules — arquivo no repo central
[submodule "repos/payments"]
    path = repos/payments
    url  = [email protected]:org/payments.git
    branch = main

[submodule "repos/identity"]
    path = repos/identity
    url  = [email protected]:org/identity.git
    branch = main

O que o Git registra

01O arquivo .gitmodules mapeia caminho → url → branch de cada submódulo.
02O repositório central guarda o SHA exato de cada submódulo — não a versão atual.
03Atualizar um submódulo é um commit no repo central. Auditável e reversível.
04Nada é copiado: os repositórios permanecem na origem, com seus donos e ciclos.

Por que submódulos
Submódulo vs alternativas

Submódulo é o caminho de menor atrito para juntar repositórios existentes.

Critério Monorepo Git Subtree Cópia manual Git Submodule ESCOLHIDO
Migração necessária Todos os repos. Por repo, irreversível. Nenhuma — mas perde-se versão. Nenhuma. Os repos ficam onde estão.
Versionamento Implícito no branch. Implícito após merge. Inexistente. Explícito. SHA por submódulo.
Isolamento de releases Difícil — tudo se move junto. Parcial. Total. Total. Cada repo mantém ciclo próprio.
Reversibilidade Quase nula. Difícil. Trivial mas frágil. Trivial. Remover é uma linha.
Adoção incremental Inviável. Por repo, irreversível. Possível. Por repo, reversível.

02Solução
Anatomia do workspace
Estrutura do repo central

Os agentes no centro, os submódulos sob repos/

# agents-workspace/
.gitmodules           # mapa dos submódulos

repos/
├── payments/        # org/payments
├── identity/        # org/identity
├── catalog/         # org/catalog
├── billing/         # org/billing
└── webapp/          # org/webapp

agents/              # personas de IA
prompts/             # prompts versionados
decisions/           # ADRs versionados

Convenções

01Submódulos sob repos/ apenas. Nada mais é referenciado externamente.
02Cada submódulo aponta para main por padrão — pinning explícito quando precisamos de estabilidade.
03O resto do workspace é nativo — não há submódulos para agentes, prompts ou docs. São arquivos do próprio repo central.
04Atualizar um submódulo é um commit. Auditável, revisível, reversível.

03Como funciona
Adicionar um submódulo
Fluxo · adicionar

Adicionar um repositório ao workspace: três comandos.

# dentro do repo central
$ git submodule add
    org/payments.git
    repos/payments
Cloning into 'repos/payments'...
done.

$ git add .gitmodules repos/payments

$ git commit -m
    "add payments submodule"
[main 8b3c19a] add payments submodule

O que acontece

01submodule add clona o repo dentro de repos/payments e registra a entrada em .gitmodules.
02Dois arquivos ficam para commit: o próprio .gitmodules e o gitlink apontando para o SHA atual.
03A partir daqui, o workspace sabe exatamente em qual versão de payments os agentes estão olhando.

03Como funciona
Clonar e atualizar
Fluxo · clonar e sincronizar

Clonar trazendo tudo, ou atualizar incrementalmente.

# clone com submódulos juntos
$ git clone --recurse-submodules
    org/agents-workspace.git

# avançar para o HEAD remoto
$ git submodule update
    --remote --merge
Updated repos/payments  a1b2 → c3d4
Updated repos/identity  e5f6 → 7a8b

# registrar a nova foto
$ git commit -am "sync"

Notas

01--recurse-submodules evita o erro mais comum: esquecer os submódulos no clone.
02--remote --merge traz o HEAD da branch configurada — sem ele, o submódulo fica preso ao SHA antigo.
03A atualização é um commit no workspace. Revisável em PR, igual a qualquer mudança.
04Para um repo específico: git submodule update --remote repos/payments

Por que agentes ganham
Seis vantagens que só esse padrão oferece

Submódulos + repo central
= contexto sólido para IA.

01

Visão unificada.

O agente abre um único repo e enxerga todo o ecossistema — sem orquestrar N clones manualmente.

04

Versão exata e reproduzível.

Cada análise tem um SHA por submódulo. A mesma pergunta sempre olha para o mesmo código.

02

Zero migração para os times.

Os repositórios continuam onde estão. Donos, PRs e CI permanecem inalterados.

05

Prompts e agentes versionados juntos.

O comportamento da IA evolui no mesmo Git que o código que ela analisa — auditável em PR.

03

Atualização explícita.

O workspace só avança quando alguém faz commit. Nada de “a IA respondeu sobre uma versão que não existe mais.”

06

Reversível e incremental.

Comece com dois submódulos. Cresça quando fizer sentido. Remova com uma linha se não funcionar.

03Como funciona
O que os agentes fazem
Casos de uso reais sobre os submódulos

Com o ecossistema reunido,
a IA faz o que antes era humanamente caro.

@reviewer

Review cross-repo

Avalia um PR em payments considerando contratos consumidos por billing e webapp.

@discoverer

Descoberta de contexto

“Onde billing depende de eventos emitidos por payments?” — responde com arquivos, linhas e ADRs relevantes.

@documenter

Documentação derivada

Mantém docs do ecossistema em sincronia com o código real, atualizando quando submódulos avançam.

@contract-watcher

Quebra de contrato

Detecta mudanças em APIs e eventos públicos antes do merge, identificando consumidores afetados nos outros submódulos.

@onboarder

Onboarding de novos engenheiros

Responde perguntas sobre o ecossistema inteiro com referências precisas — substitui a sessão 1-a-1 inicial.

@auditor

Auditoria contínua

Compara o workspace contra políticas declaradas e gera evidência rastreável para revisão e compliance.

04Operação
Armadilhas comuns
O que dá errado, e como evitar

Submódulos têm fama de difíceis.
A dor é convencional, não técnica.

01
Clonar sem --recurse-submodules
Diretórios vazios sob repos/. Nada funciona.
→ mitigar: submodule.recurse=true no clone.
04
HEAD detached em submódulo
Commits dentro de submódulos sem branch são facilmente perdidos.
→ mitigar: PRs sempre no repo de origem.
02
Esquecer de commitar o gitlink
Após submodule update, o ponteiro novo só vale após commit.
→ mitigar: CI falha se houver gitlinks pendentes.
05
Submódulo em repo privado
URLs SSH em .gitmodules quebram em CI sem credencial.
→ mitigar: deploy keys dedicadas na CI.
03
Submódulo desatualizado
Agentes acabam analisando uma foto antiga.
→ mitigar: job semanal de update --remote.
06
Crescimento sem critério
Adicionar todo repo da empresa vira ruído.
→ mitigar: apenas repos sob revisão ativa.
